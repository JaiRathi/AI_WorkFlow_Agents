2. Technology Stack & Decisions

🔤 Embedding Model

BAAI/bge-base-en-v1.5
768-dimensional embeddings. #1 on MTEB retrieval benchmark for open-source models. Handles both code and natural language. Runs on CPU (~1–2 sec per chunk batch of 32). MIT license.

🗄️ Vector Database

Qdrant (Rust, Apache 2.0)
Only open-source vector DB with native hybrid search (dense vectors + BM25 sparse in a single query). Single binary deployment. 200MB RAM baseline. Collection-level isolation per source.

⚙️ Backend

FastAPI (Python)
Async, production-grade API with auto-generated OpenAPI docs at /docs. Handles /ask, /ingest, /health endpoints with Pydantic validation. CORS-enabled for frontend.

🖥️ Frontend

Streamlit
Chat UI with message history, source expanders showing citations, index stats dashboard, and source filter controls. Python-only — zero JavaScript code.

🧠 LLM Generation

Multi-provider: OpenAI / Anthropic / Ollama
GPT-4o, Claude, or fully local Llama3 via Ollama. System prompt forces citation usage and prevents hallucination. ~3,000 tokens/query (context + question).

🎫 JIRA Integration

atlassian-python-api (Cloud)
JQL → paginated fetch → strip markup → chunk as prose. Stores jira_key, status, assignee, labels as filterable metadata in Qdrant payload.

3. Chunking Strategy (Per Source Type)

Code (.py, .ts, .java)CodeParser — AST-aware for Python, regex for others600 tokens100Split at function/class/method boundaries. Never cuts mid-function. Falls back to RecursiveCharacterTextSplitter with language-specific separators.
Test Cases (CSV/XLSX)SpreadsheetParser — csv.DictReader + pandas ExcelFile1 row = 1 chunk0Each row becomes a document. All columns formatted as key: value pairs. Metadata: test_id, module, priority.
JIRA TicketsJiraConnector — Live JQL fetch or JSON file500 tokens50Summary + description + comments concatenated. Strips Atlassian markup. Stored as prose with rich metadata payload.
PDFs / PRDs / BRDsPdfParser — PyPDF2 page extraction + python-docx500 tokens50RecursiveCharacterTextSplitter with \n\n → \n → . separators. Page number preserved in payload.
Meeting TranscriptsTranscriptParser — .txt, .vtt, .srt, .md800 tokens100VTT/SRT: strips timestamps, joins dialogue. Larger sliding window preserves conversational context across turns.
Company Docs (.md)TranscriptParser500 tokens50Split by ## and ### headings, then by paragraph. Preserves document structure.
Jenkins LogsLogParser — timestamp/error-stack aware300 tokens0Split at timestamp boundaries, ERROR/WARN/FAIL markers, or section delimiters (====). ANSI escape codes stripped.
Lucid ChartsTranscriptParser300 tokens0Each exported text block = 1 chunk. No overlap needed for discrete diagram data.

4. Hybrid Retrieval Pipeline

  1. Query embedding — User question → BGE-base-en-v1.5 → 768-dim dense vector
  2. Dense search — Cosine similarity across all collections. Top-k results scored.
  3. Parallel sparse search — BM25 keyword match (via Qdrant native fastembed) across same collections for exact term hits (function names, JIRA IDs, error codes).
  4. Reciprocal Rank Fusion (RRF) — Merges dense and sparse result sets with configurable weighting (default: 60% dense / 40% sparse). Eliminates duplicates and re-ranks.
  5. Optional re-rank — BGE-reranker-base cross-encoder refines top-20 → top-5 for maximum precision (off by default to save compute).
  6. Context assembly — Top-k chunks formatted with --- Source [filename::symbol](Lx-Ly) --- headers.
  7. LLM generation — System prompt enforces citations and prevents hallucination. Answer returned with structured source list.

8. Token Efficiency

For a typical query: Qdrant returns top-5 chunks (~500 tokens each) = ~2,500 tokens context. Add the system prompt + user question = ~3,000 tokens total per query. No entire documents are ever sent to the LLM. At GPT-4o pricing (~$2.50/1M input tokens, ~$10/1M output), each query costs ~$0.007. With Ollama + local Llama3, the cost is $0.

Key Design Decisions

  • Why Qdrant over Chroma/Weaviate/Milvus: Native hybrid search in a single query — no separate BM25 index. Single Go binary, Apache 2.0.
  • Why BGE over all-MiniLM: Higher MTEB retrieval score (63.3 vs 58.9), 768-dim gives better semantic resolution for code + prose mixing.
  • Why AST-aware code chunking: Ensures complete function/class units. Never splits a method signature from its body.
  • Why per-source collections: Enables source-type filtering at query time, per-collection stats, and independent re-indexing without affecting other sources.
  • Why 60/40 dense/sparse weighting: Semantic meaning dominates for QA questions, but keyword match catches exact JIRA keys, function names, and error codes.
